# SayIt Next Development Roadmap

> 日期：2026-06-28
> 状态：PLAN_REVIEW
> 前置条件：Round 9.1 用户实机验收通过

## 总原则

下一阶段不要同时开发：

```text
发布安装
自动升级
微信登录
用户群
场景化写作
个人表达学习
```

正确顺序是：

```text
实机稳定验收
→ 固定稳定版本
→ 对外发布基础
→ 账号与微信登录
→ 商业化授权
→ 场景化写作与个人化
```

## Stage 3 — Release Foundation（下一轮开发）

新分支：

```text
feature/release-foundation
```

执行工具：

```text
ZCode GUI → Claude Code
```

本阶段目标是让普通用户能下载安装和获得更新，不做微信登录。

### 3.1 正式安装包

- 普通用户不安装Python、Node、源码或Bridge；
- 生成Windows安装程序；
- 安装、卸载、快捷方式、单实例；
- 升级保留用户配置、数据库、词典和历史；
- 开发配置与正式用户配置彻底分离；
- 正式包不得包含开发调试入口和真实密钥。

### 3.2 统一版本系统

建立单一版本来源：

```text
app_version
backend_version
native_helper_version
config_schema_version
```

客户端“关于”页能展示完整运行版本和组件版本。

### 3.3 手动检查更新

第一轮只做：

```text
设置页 → 检查更新
```

- 获取远端签名版本清单；
- 比较版本；
- 显示更新说明；
- 下载正式安装包；
- 校验哈希/签名；
- 由用户确认安装；
- 更新失败不影响当前版本使用。

暂不做后台强制安装。

### 3.4 远程社群入口

在设置页/关于页增加：

```text
加入用户群
联系客服
```

- 客户端从远端配置读取当前二维码；
- 二维码失效可在运营端替换，不需要发新版；
- 群满时回退到客服微信/公众号；
- 不自动操作用户微信。

### 3.5 隐私和日志收口

发布前必须：

- 默认日志不得记录完整口述正文；
- 默认日志不得输出API key片段；
- crash report脱敏；
- 清楚区分本地数据与上传数据；
- 提供“打开数据目录、导出、清理”入口；
- 编写隐私说明和本地数据说明。

### 3.6 Release Foundation验收

- 全新Windows账户安装；
- 覆盖安装升级；
- 卸载后用户数据策略符合设计；
- 更新失败/断网/下载损坏；
- 不影响热键、ASR、AI和注入；
- 安装包不携带开发密钥；
- 至少一台非开发机完整测试。

## Stage 4 — Account Foundation + 微信登录

新分支：

```text
feature/account-wechat-login
```

执行工具：

```text
ZCode GUI → Claude Code
```

如果涉及登录页面和设置页面视觉设计，可先使用Figma设计，再实现。

本阶段前必须先确认：

- 使用哪个主体申请微信开放能力；
- 是否已有公司/个体主体；
- 登录用于什么权益；
- 游客是否可继续使用基础输入；
- 服务部署位置和成本；
- 隐私政策与用户协议。

### 推荐第一版账号范围

- 微信扫码登录；
- SayIt用户ID；
- 设备列表；
- refresh token撤销；
- 登录/退出；
- 登录用户获取社群入口和版本通道；
- 基础输入功能保持本地可用。

第一版不要上传：

```text
录音
完整输入正文
本地历史
个人词典
API key
```

### 第一版不建议强制登录

建议：

```text
游客可使用基础本地语音输入
登录后获得账号、设备、社群和后续授权权益
```

避免账号服务器异常时整个输入法不可用。

## Stage 5 — 商业化基础

新分支另开，不与登录混合。

可能包括：

- 试用期；
- 授权状态；
- 设备数量；
- 套餐和订阅；
- 功能开关；
- 支付与订单；
- 用户支持与反馈。

在真实用户测试前，不急着开发复杂计费。
